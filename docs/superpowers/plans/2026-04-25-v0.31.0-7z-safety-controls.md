# v0.31.0 — `.7z` archive viewer safety controls + system-resource snapshot

**Date drafted:** 2026-04-25
**Drafted by:** Claude during v0.31.0 plan execution, in response to
the user asking for headless robustness around the new `.7z` viewer
support. See parent plan: `2026-04-24-v0.31-deferred-items.md`.
**Status:** In-progress (this document captures the scope; the
implementation is happening in the same session).

## Context

The v0.31.0 deferred-items plan (Item 5 + the user's mid-session
request to make `.7z` archives readable in the log viewer) added
subprocess-based `.7z` decompression to the log search code path.
The user surfaced two correct concerns:

1. **Headless robustness** — this is unattended operation; a hung
   `7z` subprocess could pin a worker thread indefinitely.
2. **No operator visibility** — the existing 200 MB per-reader cap
   was a hardcoded magic number; operators couldn't tune it for
   their hardware, and they got no feedback while a slow search ran.

The user asked: would the following four pieces be best practice?

1. User-selectable cap with warning above a practical threshold
2. Static system-resource snapshot shown alongside the cap
3. Live progress indicator on the log viewer
4. 24h system-spec polling driving dynamic ETA estimates

I responded that #1 + #2 + #3 are appropriately scoped to v0.31.0
but #4 (the dynamic ETA framework) deserves its own separate
release because the underlying spec-polling + benchmark scaffolding
has broader applicability (bulk job ETAs, OCR queue forecasts,
etc.). User accepted that scoping. This document captures the
scoped plan.

## Item A — User-selectable `.7z` byte cap (DB-pref-driven)

**Goal:** Replace the hardcoded `_SEVENZ_MAX_BYTES = 200 * 1024 *
1024` constant with a DB preference the operator can change from
the Log Management Settings card. Show a warning when the value
crosses a practical threshold relative to host headroom.

**Backend (`core/log_manager.py`):**
- New constants:
  - `PREF_SEVEN_Z_MAX_MB = "log_seven_z_max_mb"`
  - `DEFAULT_SEVEN_Z_MAX_MB = 200`
  - `SEVEN_Z_WARN_THRESHOLD_MB = 1024` — UI shows orange warning
  - `SEVEN_Z_HARD_MAX_MB = 4096` — backend rejects above this
- New helper `get_seven_z_max_mb()` reads + clamps.
- `get_settings()` returns the new field plus `system` snapshot.
- `set_settings()` accepts the new field, writes through
  `db_write_with_retry` like other prefs.

**Backend (`api/routes/log_management.py`):**
- `SettingsUpdate` Pydantic model gains `seven_z_max_mb`.
- `_SevenZReader.__init__(path, max_bytes=None)` accepts an
  optional cap; falls back to `_SEVENZ_DEFAULT_MAX_BYTES` if not
  passed (defense-in-depth).
- The search endpoint `_do_search` reads the pref before opening
  the reader and passes it through.

**Frontend (`static/log-management.html`):**
- Settings card gains a `<input type="number">` for the cap
  (range 1-4096, step 10).
- Inline JS validation: above `SEVEN_Z_WARN_THRESHOLD_MB` a
  visible orange warning shows below the input ("This is large
  relative to typical archive sizes — set a value > free RAM
  only if you're certain your host has the headroom").

## Item B — Static system-resource snapshot

**Goal:** Show the operator current host resources (free RAM,
total RAM, CPU model, current 1-min load) alongside the byte cap
so they can size it sensibly. One-shot read at page load — no
new scheduler job, no historical tracking.

**Backend (`core/log_manager.py`):**
- New helper `get_system_resource_snapshot()`. Reads
  `/proc/meminfo` for memory, `/proc/cpuinfo` for the CPU model
  string, `os.getloadavg()` for load. Each field is best-effort
  (returns None on failure rather than raising). Linux-specific
  paths since the container is Debian.
- Returned as a sub-dict in `get_settings()` under key `system`.

**Frontend (`static/log-management.html`):**
- New row in the Settings card: "Host: <CPU model>, X GB RAM
  total / Y GB free, load <Z>". Updates on each page load /
  refresh; no polling.
- Visual cue: if `seven_z_max_mb > memory_mb_free * 0.5` show a
  warning ("Cap exceeds half of currently-free RAM").

## Item C — Live progress indicator on log viewer

**Goal:** When a search is in flight (especially `.7z`), show the
operator that work is happening, with rolling progress data.

**Backend (`api/routes/log_management.py`):**
- The current `/api/logs/search` is non-streaming — it returns
  one JSON blob with `lines` + `scanned_lines` + `wall_seconds`.
- Two implementation options:
  - **Option C1 (lightweight, chosen):** No protocol change. The
    frontend shows a busy spinner + the elapsed time since
    request started, ticking each 200ms client-side. When the
    response lands, the spinner clears and the existing status
    line populates with `scanned_lines`, `wall_seconds`, etc.
  - Option C2 (heavier, deferred): convert the search to SSE so
    the server can push progress events every N lines. Bigger
    refactor; not needed for v0.31.0 since C1 covers the
    user-visible need.
- We'll ship C1.

**Frontend (`static/log-viewer.html`):**
- Replace the current `statusEl.textContent = 'Searching...'`
  with a busy indicator that shows the rolling elapsed time
  while the request is pending.
- After response, the existing logic that sets the status line
  with returned/scanned/wall_seconds runs unchanged.
- Add a small CSS spinner to the status area.

## Item D — Deferred: 24h spec-polling + dynamic ETA framework

**Status: NOT IN v0.31.0.** Saved as a follow-up.

The full ETA framework would need:
- A `system_specs_history` table or scheduler-job preference
  storing snapshots over time
- A 24h poll job (would be scheduler job #19)
- A benchmark routine on first start (decompress a known fixed
  payload to measure bytes/sec for `.7z` and `.gz`)
- An ETA estimation function combining current load × historical
  throughput
- UI surfaces on the log viewer ("estimated 18 seconds remaining")
  AND likely the bulk job page, OCR queue page, etc. — to
  justify the framework's existence

Spinning that up properly is a separate brainstorm + plan + ship
cycle. Memory note `project_post_v031_followups.md` already
captures the broader v0.31.x followup queue; this can fold into
that bucket.

## Acceptance

- Settings page shows a 7z byte cap input, populated with the
  current pref value.
- Setting the cap above 1024 MB shows an orange warning.
- Setting the cap above 4096 MB returns 400 from the API.
- A page refresh after Save persists the value.
- The system snapshot row shows CPU model, total RAM, free RAM,
  load — all from `/proc` reads.
- Searching a `.7z` shows a spinner + ticking elapsed time
  while in flight.
- Reducing the cap below the file size truncates the search
  with `reader_warning: "7z stream truncated at N MB"` shown in
  the status line (existing surfacing — no UI change needed).
