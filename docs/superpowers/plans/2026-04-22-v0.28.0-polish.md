# v0.28.0 → v0.29.0 Polish Plan — 2026-04-22

**Context.** v0.28.0 Universal Storage Manager shipped earlier today. 83 unit + 9
integration tests pass. Branch: `feat/universal-storage-manager`. Known
follow-ups were recorded in `docs/version-history.md`. This plan executes
them in a single autonomous run with maximum parallelism.

**Autonomy ground rules.**

- Reversible, local actions only. No `git push`, no destructive operations
  on shared state.
- Commit at natural boundaries with descriptive messages.
- Run tests (in Docker via `docker run --rm`) after each workstream that
  touches backend or test code.
- Rebuild the app image before any Docker-backed verification that needs
  the latest code.
- Apple Silicon (M4): `docker run --rm` directly bypasses the NVIDIA GPU
  deploy block; do NOT use `docker-compose up` for verification.
- Skip the MCP-server meta-instructions at the top of the /effort message —
  DirectBooker, Microsoft Learn, and Learning Commons tools are irrelevant
  to this codebase work.

**Version target.** v0.28.1 on completion (polish release — no new
surface, just refinement + migration). If the Cloud Prefetch migration
(Workstream B) lands cleanly, bump to v0.29.0 instead.

## Workstream A — Modal UI polish (parallel with D, E)

**Goal.** Replace the `prompt()` / `confirm()` chains in `static/js/storage.js`
with proper modals. The current flow is functional but breaks in non-
desktop browsers and can't show validation feedback mid-form.

### A1. Add-Share modal

- Add `<div id="add-share-modal" class="modal" hidden>` to `static/storage.html`
- Form fields: name, protocol (radio: smb/nfsv3/nfsv4), server, share_path,
  username (shown only when protocol=smb), password (shown only when
  protocol=smb), read_only checkbox
- Inline validation: name must match `^[A-Za-z0-9_-]+$`; server required;
  share_path required; SMB requires username.
- Submit button disabled while an `/api/storage/shares` POST is in flight.
- Success: close modal, refresh shares table.
- Error: surface response detail inline (not via `alert()`).

### A2. Discovery modal

- One modal, two tabs: "Scan subnet" and "Probe server".
- Scan subnet: CIDR input, default populated from `/api/storage/host-info`
  first drive-letter or heuristic (not critical if empty).
- Probe server: server input + protocol radio (smb/nfs).
- Results render as a clickable list; clicking a result opens A1's Add-Share
  modal pre-filled with that server + share name.

### A3. Host-OS override dropdown

- Add a small control in the Storage page header (or inside Quick Access
  section) showing detected OS + a "Override" link.
- Override opens a dropdown: Auto / Windows / WSL / macOS / Linux.
- Persists via `PUT /api/preferences` with `host_os_override`.
- Refreshes `/api/storage/host-info` after change so Quick Access
  re-renders with the new OS's quick-access entries.

### A4. Folder-picker integration

- `static/js/folder-picker.js` already exists and is used by other pages.
- Integrate it on source-path-input and output-path-input so users can
  browse for a path instead of typing. Typing still works.
- Keep prompt-based fallback if folder-picker initialization fails.

**Verification.**
- `docker run --rm` image rebuild, hit `/storage.html`, confirm modals
  open, form validation fires, submit hits API.
- Playwright / browser automation not available in this env; manual-looking
  verification via `curl` to the API endpoints the UI hits.

**Commit (A).** `feat(storage): modal forms replace prompt() chains + host-OS override + folder-picker`

## Workstream D — Test hardening (parallel with A, B)

**Goal.** Add integration tests for error paths and role boundaries.

### D1. Role enforcement tests

- Test: POST /api/storage/shares as SEARCH_USER → 403.
- Test: GET /api/storage/shares/{name}/credentials as MANAGER → 403 (admin-only).
- Requires overriding the DEV_BYPASS_AUTH user role via a monkeypatched
  auth dependency, or by emitting a signed JWT with the target role.

### D2. Path-safety tests

- Test: POST /api/storage/validate with `/etc/passwd` → returns ok=true
  (file exists) but rejects with "file, not a folder".
- Test: POST /api/storage/validate with `/../../etc` → handled gracefully.
- Test: POST /api/storage/sources with nested paths that conflict with
  output → accepted at add time (validator doesn't check conflict), but
  the conflict check helper rejects same.

### D3. Idempotency tests

- Test: add same exclusion twice → second call creates a new entry
  (documenting current behavior, not a bug).
- Test: delete non-existent source → 204 (idempotent).

### D4. Mount manager singleton reset

- Fix: the `client` fixture in `test_storage_api.py` doesn't reset the
  MountManager singleton. Add a `monkeypatch.setattr("core.mount_manager._singleton", None)`
  in the fixture so tests don't share state.

### D5. Write-guard in practice

- Integration test: configure output path, call an internal write helper,
  confirm a write OUTSIDE the configured dir raises StorageWriteDenied.
- Uses a minimal invocation of the conversion pipeline or a direct
  `is_write_allowed()` probe.

**Commit (D).** `test(storage): role enforcement, path-safety, singleton isolation, write-guard in-practice`

## Stop-and-ask triggers

- A test newly fails that I can't quickly root-cause in under ~5 rounds
  of diagnosis.
- A `docker-compose build` fails mysteriously.
- I discover an existing bug that's orthogonal to this plan and fixing
  it would require a scope decision.
- An edit would change the Docker security posture (`SYS_ADMIN`,
  mount roots).

In those cases: stop, write a short summary, leave the branch in a
commitable state, wait for human input.
