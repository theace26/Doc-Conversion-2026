## v0.31.2 — Multi-provider 5-layer resilience port

**TL;DR:** v0.29.9 shipped a five-layer resilience pipeline for
Anthropic vision calls (preflight validation, exponential backoff
with Retry-After, per-image bisection on 400, circuit breaker,
operator banner). v0.31.0 ported the *filename interleaving*
prompt improvement to OpenAI / Gemini / Ollama. The resilience
machinery is still Anthropic-only — operators using the other
providers don't get the same protection against wasted API spend
on transient upstream failures.

**Goal:** Port the same five layers to `_batch_openai`,
`_batch_gemini`, `_batch_ollama` in `core/vision_adapter.py`.

**Notes:**
- The helpers (`_parse_retry_after`, `_backoff_delay`,
  `_safe_body`) and the circuit breaker module are already
  provider-agnostic — no need to refactor.
- Each provider's batch function needs a wrapping
  `_<provider>_sub_batch` helper analogous to
  `_anthropic_sub_batch` that handles the retry/bisect/breaker
  state machine.
- Each provider has slightly different error semantics:
  - OpenAI: 429s carry their own retry headers; per-image 400
    bisection works the same way (split content array)
  - Gemini: error responses are JSON with a `code` field; need
    a tiny extractor for the equivalent of `body[:500]`
  - Ollama: local model — circuit breaker mostly defends
    against the model crashing or running out of VRAM rather
    than rate-limiting; 4xx is rare.
- Preflight + breaker apply to all three; backoff + bisection
  apply to all three; the operator banner already polls
  `/api/analysis/circuit-breaker` which is provider-agnostic.

**Files:** `core/vision_adapter.py` (rewrite three batch
functions; ~400 LOC of additions). No frontend changes.

**Acceptance:** Switch active provider to OpenAI, run an
analysis batch with one deliberately corrupt image — verify the
batch completes for the other 9 files instead of failing all 10.
Trigger a 429-style error (use a paused / quota-exhausted key) —
verify the breaker opens, the banner appears, and operators can
reset.

**Estimated:** ~3 h, ~400 LOC, ~15k tokens.

## v0.31.3 — Preview format expansion (HEIC + RAW + SVG)

**TL;DR:** The hover preview on Batch Management currently
covers all PIL-readable formats (37 extensions across browser-
native and PIL-thumbnailed). Three categories are still
deferred:
- **HEIC / HEIF** — modern phone camera default. Need
  `pillow-heif` pip dep (~3 MB).
- **RAW camera formats** (.cr2, .nef, .arw, .raf, .dng, etc.)
  — need `rawpy` (LibRaw bindings, ~5 MB).
- **SVG** — vector format. The risk is XSS via embedded `<script>`
  / event handlers / external `<image>` references. Needs an
  XSS sanitizer (consider `bleach` or `defusedxml` + a strict
  allow-list of tags).

**Suggested split:**
- v0.31.3a — HEIC / HEIF (smallest, cleanest dep)
- v0.31.3b — RAW (bigger dep, narrower audience)
- v0.31.3c — SVG (no dep, but requires security review of the
  sanitizer)

**Files:** `requirements.txt`, `Dockerfile.base` (HEIC needs
`libheif-dev` apt pkg → triggers a base rebuild), `api/routes/
analysis.py` (extend `is_image_extension`), `core/vision_adapter.py`
(extend `_VISION_ALLOWED_MIMES`).

**Estimated:** ~2 h per sub-version, ~150 LOC each.

## v0.31.4 — Server-side ZIP bulk download

**TL;DR:** The Batch Management page's multi-file download
(v0.29.6) does sequential per-file downloads with a 120 ms
stagger. Browsers prompt once for "allow multiple downloads"
and then batch the rest, but the user's browser download
manager visibly receives N separate items. Hard cap of 100
files per trigger.

A **server-side ZIP endpoint** would stream a single download
containing all selected files. Pros: one item in the download
manager, no browser per-file prompts, server-side compression.
Cons: large ZIPs take many seconds; need a progress indicator
(WebSocket or chunked response with byte-counter polling).

**Files:** `api/routes/analysis.py` (new
`POST /api/analysis/files/download-bundle` accepting `file_ids`
+ optional `bundle_name`), `static/batch-management.html`
(replace the sequential-loop call site with a single fetch +
download).

**Streaming considerations:** Use `zipfile.ZipFile` over a
`BytesIO` buffer for small bundles, or `streamingzip` /
`zipstream` for large ones. The existing log-management bundle
endpoint already does the latter — copy that pattern.

**Estimated:** ~3 h, ~250 LOC. Out-of-scope from v0.31 per the
original plan; revisit once .7z safety + multi-provider
resilience are out of the way.

## v0.31.5 — System spec polling + dynamic ETA framework

**TL;DR:** The deferred-from-v0.31.1 piece. Build a generally-useful
ETA framework that the .7z viewer (and bulk jobs / OCR queue /
log searches) can all draw on for "this will take ~X seconds"
estimates.

**Components:**
- New scheduler job (#19): `system_spec_snapshot` runs every 24
  hours. Captures CPU, RAM, disk, GPU presence, current
  load average. Stored in a new `system_specs_history` table
  (date, snapshot_json).
- New benchmark routine on first start: decompresses a known
  fixed-size payload to measure bytes/sec for `.gz` and `.7z`.
  Result stored in DB pref. Re-runs on container start (specs
  may have changed).
- New `core/eta_estimator.py` — combines current load × historical
  throughput to produce ETA estimates for known operations.
- UI surfaces: log viewer shows ETA next to in-flight progress;
  Bulk page shows ETA on Active Jobs; OCR Review queue shows
  estimated time-to-clear.

**Files:** `core/eta_estimator.py` (new), `core/scheduler.py`
(new job), `core/db/migrations.py` (new table),
`api/routes/admin.py` (expose snapshot endpoint),
`static/log-viewer.html` / `static/bulk.html` /
`static/review.html` (UI surfaces).

**Estimated:** ~6 h, ~600 LOC, ~25k tokens. Standalone release
because it touches multiple subsystems.
