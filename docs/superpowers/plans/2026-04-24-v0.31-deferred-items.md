### Item 1 — Multi-provider filename interleaving
**Status:** Anthropic done (v0.29.8); OpenAI / Gemini / Ollama still send bare image bytes.
**Estimated:** ~30 min, ~50 LOC, ~3k tokens of work.

**Goal:** Port the `Image N filename: <name>` text-block interleaving from `_batch_anthropic` to the three other provider handlers in `core/vision_adapter.py`, so operators using OpenAI/Gemini/Ollama also get filename-grounded descriptions.

**Files:**

**Per-provider implementation notes:**

  - OR (cleaner): switch Ollama to the same multi-modal format if the model supports it
  - Recommendation: filename list in prompt is the lowest-friction option

**Acceptance:**

### Item 3 — Bulk re-analyze with filter, delete-and-re-insert semantics
**Status:** Per-row endpoint exists (`POST /api/analysis/queue/{id}/reanalyze`, v0.30.4) but uses UPDATE-in-place semantics. No bulk variant.
**Estimated:** ~2 h (was 1.5 h; the semantics change adds ~30 min), ~250 LOC, ~12k tokens.

**Goal:** Let operators bulk-re-analyze every row matching a filter — primarily "all rows analyzed before {date}" so they can refresh stale results after a prompt improvement (like v0.29.8's filename context).

#### IMPORTANT — semantics change: delete-and-re-insert, not UPDATE-in-place

**Why this matters (per user requirement, 2026-04-24):**

> *"Make sure that the files that get reanalyzed are deleted from the database and resubmitted for entry into the database with the new information."*

The v0.30.4 per-row endpoint currently does an UPDATE that preserves `id`, `enqueued_at`, and other identity fields while clearing the analysis output fields. The user wants a true clean slate:

- DELETE the existing `analysis_queue` row entirely
- Re-INSERT a fresh row via `enqueue_for_analysis(source_path, content_hash, ...)`
- Result: new `id`, fresh `enqueued_at`, fresh `retry_count = 0`, no carry-over of any prior column state

This applies to BOTH the existing per-row endpoint AND the new bulk variant. **Item 3 includes fixing v0.30.4's per-row endpoint to match this semantics.**

#### Reasons delete-and-re-insert is the right model
- No risk of stale columns carrying over (description, extracted_text, error, retry_count all start NULL/0 because the row is brand-new, not because we explicitly cleared them — defends against future schema additions that the UPDATE would have to remember to clear)
- Fresh `id` means any external system that cached the old id will get a clear "row not found" rather than a confusing "row exists but is now pending" — easier to reason about
- Matches the operator's mental model ("treat this as if scanned for the first time")
- `enqueue_for_analysis` is the canonical insertion path; reusing it ensures any future enqueue-time logic (validation, hashing, etc.) is automatically applied

#### Implementation

**Per-row fix (apply to v0.30.4 endpoint):**
```python
@router.post("/queue/{entry_id}/reanalyze")
async def reanalyze_queue_entry(entry_id: str, ...):
    # 1. Look up the existing row (need source_path + content_hash + file_category + scan_run_id + job_id)
    row = await db_fetch_one("SELECT * FROM analysis_queue WHERE id = ?", (entry_id,))
    if not row:
        raise HTTPException(404, "Analysis entry not found")

    # 2. DELETE the row
    async def _delete():
        async with get_db() as conn:
            await conn.execute("DELETE FROM analysis_queue WHERE id = ?", (entry_id,))
            await conn.commit()
    await db_write_with_retry(_delete)

    # 3. Re-insert via the canonical path
    new_id = await enqueue_for_analysis(
        source_path=row["source_path"],
        content_hash=row["content_hash"],
        job_id=row["job_id"],
        scan_run_id=row["scan_run_id"],
        file_category=row["file_category"] or "image",
    )
    return {"status": "queued", "old_entry_id": entry_id, "new_entry_id": new_id}
```

Note: `enqueue_for_analysis` has a skip-on-completed check — but the row is gone now, so it will insert fresh. The existing skip logic is therefore unaffected for OTHER files.

**Bulk endpoint:** `POST /api/analysis/queue/reanalyze-bulk` (OPERATOR+)
```python
class BulkReanalyzeRequest(BaseModel):
    analyzed_before_iso: str | None = None    # primary use case
    analyzed_after_iso: str | None = None     # symmetric
    provider_id: str | None = None            # "all rows from old provider X"
    model: str | None = None                  # "all rows from old model Y"
    status: str | None = None                 # default "completed"; can be "failed" for retry-everything
    dry_run: bool = True                      # default true — return count + sample
```

Bulk semantics — same delete-and-re-insert per matched row:
1. SELECT matching rows (capture `id`, `source_path`, `content_hash`, `job_id`, `scan_run_id`, `file_category`)
2. If `dry_run`: return `{matched: N, sample: [first 5 source_paths]}` and stop
3. Otherwise: DELETE matching rows in one statement; loop over captured tuples calling `enqueue_for_analysis(...)` for each
4. Return `{deleted: N, re_enqueued: M, new_entry_ids: [...]}` (M may be < N if some files moved or content_hash conflicts emerge — log the diff)

**Cap + safety:** 10,000 rows per call. 400 if exceeded. All writes through `db_write_with_retry`.

#### Frontend

- New "Bulk re-analyze..." button on Batch Management top bar (next to Cancel All Pending)
- Modal with:
  - Date picker (default = midnight of analyzed_at column min)
  - Optional provider/model dropdowns (pre-populated from distinct values in DB)
  - Status filter (default completed)
  - "Preview" button → calls `dry_run=true`, shows count + sample paths
  - "Re-analyze N rows" button → confirm dialog explicitly mentions: "**This deletes the matching rows and re-submits them as fresh entries.** Old `analysis_queue.id` values will be lost. Consumes ~N image-analysis worth of LLM tokens."

#### Files

- `api/routes/analysis.py` — UPDATE `reanalyze_queue_entry` to delete+re-insert; ADD `reanalyze_bulk` endpoint + Pydantic model
- `core/db/analysis.py` — new `bulk_delete_and_re_enqueue(filter_dict, dry_run)` helper that returns the captured tuples for the API to loop on
- `static/batch-management.html` — top-bar button + modal HTML/CSS/JS; update the per-row "Re-analyze" button copy in the analysis-result modal to mention "deletes and re-submits"

#### Acceptance

- Per-row "Re-analyze" returns `{old_entry_id, new_entry_id}` with different IDs
- After per-row re-analyze, querying the OLD `entry_id` returns 404
- After per-row re-analyze, the NEW row exists with `status='pending'`, `enqueued_at` is recent (within seconds), `retry_count=0`, all output columns NULL
- Dry-run bulk with `analyzed_before_iso=2026-04-24T15:00` returns matching count without modifying
- Bulk confirm dialog mentions delete-and-re-insert explicitly
- After actual bulk run: `deleted` count == matched count; `re_enqueued` count usually == `deleted`; new IDs are returned
- Worker re-submits new rows on next claim cycle

#### Risks

- Operator could blow LLM quota fast — the 10k cap + dry-run + confirm dialog are the safety net
- **External systems caching `analysis_queue.id` values lose those references after re-analyze.** Currently no known consumers do this (the analysis_queue.id is internal to the analysis subsystem; the UI fetches by id but doesn't persist it). Document behavior in `docs/gotchas.md` so future integrations know.
- A row deleted between SELECT and DELETE (race with another operator) is silently dropped from the re-enqueue loop. Acceptable; log the diff.

### Item 5 — Consolidate `core/log_archiver.py` into `core/log_manager.py`
**Status:** Two log subsystems coexist. log_archiver runs 6-hourly, gzip only, hardcoded archive subdir + 90-day retention. log_manager handles UI/admin manual triggers.
**Estimated:** ~2 h, ~100 LOC + a delete, ~10k tokens.

**Goal:** Single source of truth for log compression + retention. UI settings (compression format, retention days, rotation size) actually govern behavior — currently the automated job ignores them.

**Approach:**
1. Delete `core/log_archiver.py`
2. Add a new scheduler job in `core/scheduler.py` that calls `log_manager.compress_rotated_logs()` + `log_manager.apply_retention()` on the same 6-hourly cadence (or configurable via preference `log_archive_interval_hours`, default 6)
3. The new job respects DB preferences (compression format, retention days) — already wired in log_manager
4. Migrate the existing `/app/logs/archive/*.gz` files: leave them in place (the new manager already discovers them via the `archive/` subdir scan added in v0.30.1)
5. The new compression writes to `/app/logs/<name>.gz` (in-place rotation), NOT to the `archive/` subdir — that's the convention the new manager assumed. Existing archive files stay accessible read-only via the inventory.
6. Remove the `from core.log_archiver import archive_rotated_logs` line in scheduler.py

**Risks:**
- A running container has the existing scheduler job loaded. Replacing it means the next startup has the new job; the old in-flight job (if running) finishes its current pass and then disappears.
- Operators who depend on the `archive/` subdir convention (scripts, external monitoring) need a heads-up. Add to docs.

**Files:**
- `core/scheduler.py` — replace `archive_rotated_logs` job with new `_compress_and_retain` wrapper
- `core/log_archiver.py` — DELETE
- `core/log_manager.py` — verify `compress_rotated_logs` + `apply_retention` are scheduler-callable as is (should be — they're already async)
- `docs/gotchas.md` — add note about `archive/` subdir transition
- `docs/version-history.md` — explain the consolidation
- `CLAUDE.md` — drop log_archiver mention from the file map

**Acceptance:**
- After deploy + 5+ minutes (or trigger manually via admin button), rotated logs compress with the format set in the Settings panel
- Retention days from settings actually purge old files
- `/api/logs` inventory shows both old `archive/*.gz` files and new in-place `*.gz` files
